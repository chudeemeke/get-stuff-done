**Location:** `node_modules/get-shit-done-cc/get-shit-done/workflows/plan-phase.md` line **472** (was line 457 at v1.38.2 per RESEARCH.md §6; shifted +15 lines from other commits in v1.38.2..v1.39.1; file now 1620 lines total).

```bash
echo "$PHASE_SECTION" | grep -iE "UI|interface|frontend|component|layout|page|screen|view|form|dashboard|widget" > /dev/null 2>&1
```

**Context (lines 457-476):** This is the "5.6. UI Design Contract Gate" frontend-indicator check. Output is `HAS_UI=0` (frontend found) vs `HAS_UI=1` (no frontend) — drives whether UI-SPEC.md is required.

**Bug shape — no word boundaries:** The regex uses `grep -iE` (case-insensitive extended) with pipe-alternation of keywords WITHOUT `\b...\b` word-boundary anchors. This produces false-positive HAS_UI=0 matches on common non-UI words:

| Keyword | False-positive substring matches |
|---|---|
| `UI` | "GUIdance", "intUItion", "fluid" |
| `form` | "perform", "format", "transform", "information", "platform", "conform" |
| `view` | "preview", "overview", "interview", "review" |
| `page` | "pagestate", "homepagecheck" |
| `screen` | "screening" |
| `view` | "viewpoint" |

The false-positive rate matters because the gate's failure mode is **noisy** (requires UI-SPEC.md when the phase has no UI work), forcing users into `--skip-ui` workarounds or spurious UI-SPEC.md authoring.

**Recommended upstream fix:** `grep -iwE "..."` (`-w` adds word boundaries) OR explicit `\b...\b` anchoring in the pattern.

**Disposition per D-20:** file upstream issue + PR in Wave 5 (Phase 40.5's final wave). 999.3 is NOT closed by upstream at v1.39.1 — bug remains. Wave 5 task: open PR against `gsd-build/get-shit-done` (upstream repo confirmed from `node_modules/get-shit-done-cc/package.json` `repository.url`) with the `-w` fix; reference the false-positive substrings table in the PR body.
