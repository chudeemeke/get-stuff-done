### Critical: gsd-tools.js validation require breaks post-install

`get-stuff-done/bin/gsd-tools.js` line 37:
```javascript
const { validateGitSHA, ... } = require('../../src/validation');
```

- **At source:** `../../src/validation` resolves to `<project>/src/validation/` -- works
- **Post-install:** `~/.claude/get-stuff-done/bin/gsd-tools.js` -> `~/.claude/src/validation/` -- DOES NOT EXIST
- **Impact:** MODULE_NOT_FOUND on every GSD command that invokes gsd-tools.js
- **Why tests pass:** Tests run from project root where relative path resolves correctly
- **Precedent:** Same class of bug Phase 13 fixed for hooks via esbuild bundling
- **Fix options:** (A) esbuild bundle gsd-tools.js like hooks, (B) installer copies src/ alongside get-stuff-done/, (C) inline validators into gsd-tools.js

## Gaps

### Critical (blocks ship)

**GAP-1: gsd-tools.js validation require path breaks post-install**

- `require('../../src/validation')` resolves at source but fails after installer copies to `~/.claude/`
- The installer copies `get-stuff-done/` but not `src/` to the install destination
- Every GSD command using gsd-tools.js would fail with MODULE_NOT_FOUND
- Same bug class as Phase 13 (hooks), same fix pattern available (esbuild bundling)
- **Affected files:** `get-stuff-done/bin/gsd-tools.js` (line 37), `bin/install.js` (copy logic)
- **Fix pattern:** esbuild bundle gsd-tools.js to inline src/validation (Phase 13 precedent), or have installer copy src/ to ~/.claude/src/, or inline validators into gsd-tools.js

**GAP-2: macOS/Linux manual testing not performed**

- Phase 9 cross-platform code is complete but never tested on physical macOS or Linux hardware
- CI matrix (Phase 11) runs on all 3 platforms and passes, covering functional correctness
- Manual testing needed for: installation (symlinks created correctly), launcher execution, hook execution, statusline rendering with Unicode/ANSI, permission failure fallback chain
- **Affected files:** `bin/install.js` (symlink/junction/copy fallback), `bin/gsd.js` (launcher), `hooks/pre-compact.js`, `hooks/gsd-statusline.js`
- **Fix pattern:** Create automated integration tests that simulate install + run on each platform, or document manual test results from CI artifacts. Extend test suite with platform-specific assertions for symlink creation, path resolution, and terminal capability detection.

**GAP-3: Platform module coverage below 95% WoW threshold**

- `src/platform/detect.js`: 100% functions, 66.67% lines -- shell detection branches (PowerShell/cmd/WSL/zsh) only execute on matching OS
- `src/platform/terminal.js`: 100% functions, 93.50% lines -- Windows Console Host detection edge case uncovered
- Overall platform modules: 93.54% functions, 90.50% lines (WoW requires 95%+ at EACH metric)
- **Affected files:** `src/platform/detect.js` (detectShell, detectEnvironment branches), `src/platform/terminal.js` (Windows Console Host path), `tests/platform.test.js`
- **Fix pattern:** Add platform-specific test cases using Object.defineProperty mocking for process.platform, process.env.SHELL, process.env.WT_SESSION. Mock terminal detection paths for each OS. Target: 95%+ lines in both detect.js and terminal.js individually.

**GAP-4: src/platform/index.js is dead code**

- Phase 9 created a unified platform API re-export (`module.exports = { ...detect, ...paths, ...terminal }`)
- Zero consumers in production code -- every file imports submodules directly (`require('./platform/paths')`, `require('./platform/terminal')`)
- Either delete the file or refactor consumers to import from the unified index
- **Affected files:** `src/platform/index.js` (dead code), `bin/gsd.js`, `hooks/pre-compact.js`, `hooks/gsd-statusline.js`, `src/config/ConfigLoader.js` (potential consumers to refactor)
- **Fix pattern:** Either (A) delete src/platform/index.js and remove from package.json files array, or (B) refactor all platform submodule imports to use the unified index. Option A is simpler and avoids unnecessary indirection.

**GAP-5: Agent teams not wired into workflows**

- Phase 10 created 4 team templates (`get-stuff-done/teams/`) and 4 oversight agents (`agents/gsd-oversight-*.md`)
- `config.json` has `teams.enabled`, `teams.oversight`, and `teams.soft_cap` settings
- No workflow reads `teams.enabled` to conditionally spawn teams instead of sequential subagents
- Team templates and oversight agents are created but never referenced by executable workflow code
- The `teams` section of config.json is also orphaned since nothing reads it
- Phase 10 success criteria says "implemented" but the feature is only "created" -- templates exist without wiring
- **Affected files:** `get-stuff-done/workflows/execute-phase.md`, `get-stuff-done/workflows/plan-phase.md`, `get-stuff-done/workflows/upstream-sync.md`, `get-stuff-done/workflows/verify-phase.md` (need conditional team spawning), `get-stuff-done/teams/*.md` (4 templates), `agents/gsd-oversight-*.md` (4 oversight agents), `.planning/config.json` (teams section)
- **Fix pattern:** Add conditional logic to 4 workflows: read `teams.enabled` from config.json, if true spawn team using template + oversight agent, if false use existing sequential subagent fallback. Each workflow already has subagent fallback sections from Phase 10 -- the missing piece is the conditional branch that reads config and routes to team mode.
