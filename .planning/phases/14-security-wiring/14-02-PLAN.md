CRITICAL: Match CONTEXT.md Decision #4 exactly. The function MUST include an explicit `process.exit(1)` call, not rely solely on error() exiting:
```javascript
function requireValid(result) {
  if (!result.ok) {
    error(result.error);
    process.exit(1);
  }
  return result.value;
}
```

Why both `error()` and `process.exit(1)`: `error()` currently calls `process.exit(1)` internally, but requireValid must be self-contained. If `error()` is ever refactored to log without exiting (e.g., for batch error collection), requireValid must still halt. This matches the CONTEXT.md locked pattern exactly.

Per CONTEXT decision: requireValid lives in gsd-tools.js (application layer), NOT in the validation module. The validation module stays pure (no process.exit knowledge).

**Step 2: Validate GSD_CONFIG_PATH in getConfigPath() using dual-base approach:**

The implementation MUST use `allowedBases = [os.homedir(), os.tmpdir()]` to support both production use (homedir) and testing (tmpdir). Do NOT use a single-base approach -- existing tests at lines 101, 116, 135, 146, 154 set GSD_CONFIG_PATH to paths in temp directories created by createTempDir(), which live in os.tmpdir(), not os.homedir().

```javascript
function getConfigPath() {
  if (process.env.GSD_CONFIG_PATH) {
    // Validate path is within user-accessible directories
    // homedir: production config paths (~/.gsd/config.json)
    // tmpdir: test harness paths (createTempDir creates under os.tmpdir)
    const allowedBases = [os.homedir(), os.tmpdir()];
    for (const base of allowedBases) {
      const result = validateConfigPath(process.env.GSD_CONFIG_PATH, base);
      if (result.ok) {
        return result.value;
      }
    }
    throw new Error('GSD_CONFIG_PATH validation failed: path must be within home or temp directory');
  }
  return path.join(os.homedir(), '.gsd', 'config.json');
}
```

This allows homedir (production) and tmpdir (testing) while rejecting /etc/shadow and other system paths.
