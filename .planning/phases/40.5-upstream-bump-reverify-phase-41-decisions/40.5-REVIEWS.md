### Diagnosis

Three different invocations (exec stdin, exec positional, exec review) all hung at the same phase: **after Codex emitted a "thinking" reasoning block describing which tool it was about to call, but before the tool call itself executed**. Stderr stops growing at exactly the moment the model should issue its first shell/read tool. The hang persists across reasoning-effort levels. Only available model (`gpt-5.4`) consistently exhibits this. Cancelling and retrying does not help.

This is **not** the long-prompt hang described in `feedback_review_cli_invocation.md`; the prompts in attempts 2-5 were short (47-50 lines) or zero (review subcommand uses commit diff only). The hang pattern is **planning→execution transition stalls**, distinct from the prior memory's "long prompt → indefinite hang" pattern.

### Recommendation

The Codex CLI is currently unusable in non-interactive mode on this Windows install for tasks beyond trivial planning. Two paths:

1. **File a bug upstream** at [openai/codex](https://github.com/openai/codex) — reproducer: codex 0.104.0 on Windows MINGW with ChatGPT-account auth (gpt-5.4 only); any `codex exec review --commit <sha>` invocation hangs at planning→execution transition. Repro is reliable across 4 invocation variants.

2. **Update the local feedback memory** `feedback_review_cli_invocation.md` to reflect that the documented stdin-pipe workaround does NOT resolve all Codex hangs — the planning→execution stall is a separate failure mode. Future sessions should not waste time re-attempting Codex review on this setup until upstream fixes the issue.

Both paths recommended. Friction-log the recurrence so any third Codex hang in another project's session triggers the inbox-promotion threshold from the cross-project-issues convention.
