       - use `gitleaks/gitleaks-action@v2` per SECURITY-02 and D-07.

       - use a normal job, not OSV's reusable workflow, because this phase must
         run `step-security/harden-runner@v2` and local triage steps in the same
         job.
       - use the direct OSV scanner action path
         `google/osv-scanner-action/osv-scanner-action@v2`, with a comment in
         the workflow noting this is intentionally chosen over the recommended
         reusable workflow for same-job harden-runner telemetry and local
         triage.
       - scan `bun.lock`.
       - produce JSON output at `osv-results.json`.
       - allow the raw scan step to continue so all severities can be triaged,
         then run `node scripts/osv-triage.js --input osv-results.json --output
         osv-triage.json --fail-on high,critical` as the blocking step.
       - route MEDIUM/LOW output through `scripts/osv-triage.js` and a GitHub
         issue/comment step with dedup by issue title.
       - do not use scanner-level filtering that hides MEDIUM/LOW findings
         before triage.

    4. Harden-runner audit mode:
       - install `step-security/harden-runner@v2` with `egress-policy: audit` in
         every new security job.
       - do not set block mode in Phase 41.
