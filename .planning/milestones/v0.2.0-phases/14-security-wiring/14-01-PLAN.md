<task type="auto">
  <name>Task 1: Migrate validation module to Result type and add new validators</name>
  <files>src/validation/index.js</files>
  <action>
Rewrite src/validation/index.js to use Result type pattern for ALL functions. This is a breaking API change -- no function should throw exceptions after this task.

**Result type pattern (per CONTEXT.md locked decision):**
- Success: `{ ok: true, value: sanitizedInput }`
- Failure: `{ ok: false, error: 'descriptive message' }`

**Migrate existing functions:**

1. `validateGitSHA(sha)` -- Keep existing regex `/^[0-9a-f]{7,40}$/i`. Return `{ ok: true, value: sha.toLowerCase() }` on success (normalize to lowercase). Return `{ ok: false, error: 'Invalid git SHA format: ...' }` on failure.

2. `validateBranchName(branch)` -- Keep existing pattern `/^[a-zA-Z0-9][a-zA-Z0-9/_.-]*$/`, max 255 chars, reject `..`, `.lock`, `@{`. Return `{ ok: true, value: branch.trim() }` on success (trim whitespace). Return `{ ok: false, error: '...' }` on failure with specific reason.

3. `validateConfigPath(userPath, allowedBaseDir)` -- Keep existing logic (decodeURIComponent, path.resolve, cross-platform slash normalization, startsWith check). Return `{ ok: true, value: resolvedPath }` on success. Return `{ ok: false, error: 'Path traversal detected: ...' }` on failure.

**Add new functions:**

4. `validateTagName(tag)` -- Similar rules to branch names per git-check-ref-format(1). Allowlist: alphanumeric start, then `a-zA-Z0-9._/-`. Reject: `..`, `.lock` suffix, `@{`, max 255 chars. Return `{ ok: true, value: tag }` (no transformation -- tags are case-sensitive).

5. `validateRemoteURL(url)` -- Protocol allowlist per CONTEXT.md locked decision:
   - Allow: `https://` (pattern: `/^https:\/\/.+/`)
   - Allow: `ssh://` URI form (pattern: `/^ssh:\/\/.+/`)
   - Allow: `git@host:path` scp-like SSH (pattern: `/^[a-zA-Z0-9._-]+@[a-zA-Z0-9._-]+:[a-zA-Z0-9._\/-]+/`)
   - Reject everything else with error: `Remote URL uses disallowed protocol: {url}. Allowed: https://, ssh://, git@host:path`
   - Return `{ ok: true, value: url }` on success (no transformation -- URLs are case-sensitive in path)

**Export all 5 functions from module.exports.**

**Important:** The validation module must remain PURE -- no knowledge of process.exit, error(), or any application-layer concerns. It is a domain module. The requireValid() bridge lives in gsd-tools.js (Plan 02).
  </action>

<task type="auto">
  <name>Task 2: Rewrite all tests for Result type and add new validator tests</name>
  <files>tests/validation.test.js</files>
  <action>
Rewrite the entire tests/validation.test.js file to use Result type assertions instead of exception-based assertions. All 33 existing tests must be migrated, and comprehensive tests must be added for the 2 new validators.

**Test migration pattern (every test changes):**

Happy path BEFORE: `expect(validateGitSHA(sha)).toBe(sha)`
Happy path AFTER: `const result = validateGitSHA(sha); expect(result.ok).toBe(true); expect(result.value).toBe(sha.toLowerCase());`

Error path BEFORE: `expect(() => validateGitSHA(sha)).toThrow('Invalid git SHA format')`
Error path AFTER: `const result = validateGitSHA(sha); expect(result.ok).toBe(false); expect(result.error).toContain('Invalid git SHA format');`

**Migrate existing test groups:**

1. `validateGitSHA` -- 8 happy-path tests (update return value to check `result.ok === true` and `result.value === sha.toLowerCase()`), 11 error-path tests (update to check `result.ok === false` and `result.error` contains expected message). Note: the lowercase normalization means `expect(result.value).toBe(sha.toLowerCase())` for happy paths.

2. `validateBranchName` -- 9 happy-path tests (update to check `result.value === branch.trim()`), 14 error-path tests (update to check `result.ok === false`).

3. `validateConfigPath` -- 3 happy-path tests (update to check `result.ok === true` and `result.value` contains tempDir), 6 error-path tests (update to check `result.ok === false`), 3 platform tests, 3 invalid input tests.

**Add new test groups:**

4. `validateTagName` -- Test cases:
   - Happy: `v1.0.0`, `release/1.0`, `my-tag`, `MyTag`, `tag.with.dots`, `tag_underscore`, long tag under 255 chars
   - Error: empty string, non-string, starts with hyphen, starts with dot, contains `..`, ends with `.lock`, contains `@{`, contains shell metacharacters (`;`, `|`, `&`, `$`, backtick), exceeds 255 chars, contains spaces, contains `~`, `^`, `:`, `?`, `*`, `\`

5. `validateRemoteURL` -- Test cases:
   - Happy: `https://github.com/user/repo.git`, `https://gitlab.com/user/repo`, `ssh://git@github.com/user/repo.git`, `git@github.com:user/repo.git`, `git@gitlab.com:group/subgroup/repo.git`
   - Error: `git://github.com/user/repo.git` (disabled by GitHub), `file:///etc/passwd` (SSRF/LFI), `http://github.com/user/repo.git` (no TLS), `ftp://example.com/repo`, empty string, non-string, `javascript:alert(1)`, random text without protocol

**Keep the existing test helper usage** (createTempDir for validateConfigPath tests).

**CRITICAL:** Zero `toThrow()` assertions should remain after migration. Every assertion should use `result.ok` and `result.value` or `result.error`.
