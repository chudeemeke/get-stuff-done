    Create tests/installer.test.js covering bin/install.js.

    **Testing strategy:** install.js is a 1,760-line CLI script that modifies the real filesystem (creates dirs, symlinks, writes settings.json). It runs as a postinstall script. Testing requires:
    1. Heavy filesystem mocking -- ALL operations target temp directories
    2. Environment isolation -- HOME, USERPROFILE, APPDATA point to temp dirs
    3. Process mock -- npm_config_global, npm_lifecycle_event control runtime detection
    4. Permission simulation -- Mock fs.accessSync to simulate permission failures

    Read bin/install.js first to identify the main exported/callable functions and the control flow. The file likely has a main() that calls helper functions. Test the helpers individually where possible, and test main() as integration with full env mocking.

    **Runtime detection tests:**
    - Detects npm global install (npm_config_global=true)
    - Detects bun global install (appropriate env markers)
    - Detects npx execution (npm_lifecycle_event checks)
    - Detects local install vs global install
    - Falls back gracefully for unknown runtime

    **Directory creation tests:**
    - Creates ~/.gsd/ directory structure
    - Creates ~/.claude/ directory structure if needed
    - Handles pre-existing directories (no error, no overwrite of user data)
    - Creates subdirectories (commands/, agents/, hooks/dist/, etc.)
    - Respects GSD_HOME override for non-standard locations

    **Symlink/junction/copy tests:**
    - Attempts symlink first on Unix
    - Falls back to junction on Windows when symlink fails
    - Falls back to copy when both symlink and junction fail
    - Records install method in metadata
    - Copy mode copies all required files (agents, commands, workflows, hooks, etc.)
    - Symlink points to correct source directory

    **Hook registration tests:**
    - Creates or updates Claude Code settings.json with hook entries
    - Adds hooks array with gsd-statusline, gsd-check-update, pre-compact
    - Preserves existing settings.json content (does not overwrite user settings)
    - Handles missing settings.json (creates new)
    - Handles malformed settings.json (error message, not crash)

    **Metadata tests:**
    - Writes .install-meta.json with install method, timestamp, source path
    - Updates metadata on reinstall

    **Error handling tests:**
    - Permission denied on mkdir -> clear error message with suggested fix
    - Permission denied on symlink -> falls back to junction/copy
    - Source directory missing -> clear error message
    - Disk full simulation -> graceful failure

    **Upgrade path tests:**
    - Detects existing installation
    - Preserves user config on upgrade
    - Updates hooks and commands on upgrade
    - Cleans up stale symlinks

    **Mocking approach:**
    - Use createTempDir from helpers for all filesystem targets
    - Set HOME (Unix) and USERPROFILE + APPDATA (Windows) to temp dirs
    - Create a mock "source" directory with expected package structure
    - Use mockEnv to control npm_config_global, npm_lifecycle_event
    - For symlink/junction failures: create read-only temp dirs or use jest-style mocking of fs.symlinkSync
    - Run installer via execSync with full env override: node bin/install.js with env pointing to temp dirs
    - Set GSD_INSTALL_TEST=1 or similar env flag if needed to prevent the installer from modifying real system dirs (check if install.js has a test mode or if env isolation is sufficient)

    **IMPORTANT:** Never let tests modify the real ~/.gsd or ~/.claude directories. All HOME/USERPROFILE env vars MUST point to temp directories. Verify this in beforeEach setup.

    All tests use bun:test. Clean up ALL temp dirs in afterEach.
