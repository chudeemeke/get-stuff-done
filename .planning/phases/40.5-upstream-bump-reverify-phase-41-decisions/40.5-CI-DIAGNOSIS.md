## Root causes (3 distinct)

### Cause 1: Upstream test bundling × bun-test discovery glob

**Failures:** ~538 (~77 test files)
**Locations:** `dist/sdk/src/**/*.test.ts`
**Sample test:** `agentSkills > returns valid QueryResult with skills array` at `dist/sdk/src/query/decomposed-handlers.test.ts:74:40`

**Mechanism:**
1. Upstream `get-shit-done-cc@1.38.5` ships **77 `.test.ts` files** bundled inside its npm package (was not the case in 1.34.2).
2. Fork's CI workflow runs `bun install && bun run compose && bun test`.
3. `bun run compose` writes to `dist/` from `node_modules/get-shit-done-cc/`.
4. `bunfig.toml` test config:
   ```toml
   [test]
   include = ["**/*.test.js"]
   exclude = ["**/*.test.cjs", "**/node_modules/**"]
   ```
   `node_modules/**` is excluded — `dist/**` is NOT.
5. Bun-test auto-discovers `dist/sdk/src/**/*.test.ts` (bun is permissive about `.ts` extension under `.test.js` glob).
6. Discovered tests run as upstream-internal tests in fork context — they need fixtures, environment, and module paths that only exist in upstream's own repo, not the fork's compose output. They fail.

**Why this didn't show locally:** Agent ran `bun test` against the project root's `dist/`, which still contained the OLD 1.34.2 compose output (zero `.test.ts` files). The freshly-composed worktree dist contains 77. Bypassing compose in local verification bypassed the failure mode entirely.

**Fix:** Add `dist/**` (and ideally `**/*.test.ts` if fork has no `.test.ts` files of its own) to bunfig.toml exclude. Confirm by running a fresh `bun install && bun run compose && bun test` locally and observing the test count drops to fork-only.

**Why this is a categorical fix, not a per-test fix:** All ~538 failures share one root cause (`dist/` not excluded). One config change resolves all of them simultaneously.

### Cause 2: Stale path references in `tests/hooks.test.js`

**Failures:** 42 (all in `tests/hooks.test.js`)
**Sample tests:**
- `overlay/hooks/gsd-statusline.js > produces output string`
- `overlay/hooks/gsd-check-update.js > creates cache directory if missing`

**Mechanism:**
At `tests/hooks.test.js:101-105`:
```js
const HOOKS = {
  checkUpdate: path.join(PROJECT_ROOT, 'overlay', 'hooks', 'gsd-check-update.js'),
  statusline: path.join(PROJECT_ROOT, 'overlay', 'hooks', 'gsd-statusline.js'),
  preCompact: path.join(PROJECT_ROOT, 'overlay', 'hooks', 'pre-compact.js'),
};
```

But on disk:
- `overlay/hooks/` contains ONLY `pre-compact.js` (and `pre-compact.sh`)
- `overrides/hooks/` contains `gsd-check-update.js` and `gsd-statusline.js` (with REASON.md files)

The test file documentation comment at line 6 reads `overlay/hooks/gsd-statusline.js` — wrong location. This test was written when the convention was different. During the v3.0.0 architecture transition, two of the three hooks moved to `overrides/hooks/` (because they're overrides of upstream files), and `pre-compact.js` stayed in `overlay/hooks/` (no upstream counterpart). The test never updated.

This is **not a bump regression** — this has been broken since the v3.0.0 transition on 2026-03-31. It just shows up now because the bump's other failures swamped the signal.

**Fix:** Update `HOOKS` constant to point checkUpdate and statusline at `overrides/hooks/`. Possibly also update line-6 comment.

**Mechanism:** Not investigated in stage 1 — this is the work that Phase 41 was scoped to address (REL-01: 100% test pass).

**Fix scope:** Hours-to-days, requires actual engineering investigation. Three are installer-related — likely one or two underlying causes. Fourth is a security assertion (path traversal rejection).

**Recommendation:** Don't address in Phase 40.5. These are Phase 41's work. Phase 40.5's redefined scope should be limited to fixes 1 and 2 + the bump itself.
