**Objective section:**
Sync commits from glittercowboy/get-shit-done, allow selection/cherry-pick, verify changes, and publish to npm with proper versioning.

**Process section - Pre-flight checks:**
1. Verify git working directory is clean (no uncommitted changes)
   - If dirty: ERROR with `git status` output, require user to commit or stash
2. Verify GitHub auth (SSH key or gh CLI)
   - Run: `git ls-remote --exit-code git@github.com:chudeemeke/get-stuff-done.git HEAD`
   - If fails: Offer to run `gh auth login`
3. Verify npm auth
   - Run: `bun pm whoami` or `npm whoami`
   - If fails: Offer to run `npm login`
4. Add upstream remote if missing
   - Check: `git config --get remote.upstream.url`
   - If missing: `git remote add upstream https://github.com/glittercowboy/get-shit-done.git`

**Process section - Spawn workflow:**
After pre-flight passes, spawn the upstream-sync workflow:

```markdown
Task(
  prompt="First, read ~/.claude/get-stuff-done/workflows/upstream-sync.md for workflow instructions.\n\n<sync_context>\n{cache_json_content}\n</sync_context>",
  subagent_type="general-purpose",
  model="opus",
  description="Upstream sync workflow"
)
```

**Process section - Handle workflow returns:**
- `## SYNC COMPLETE` - Display summary, update cache, show success
- `## CHECKPOINT: *` - Present to user, collect response, spawn continuation
- `## SYNC ABORTED` - Show reason, no changes made
- `## CONFLICT DETECTED` - Show conflicted files, wait for manual resolution

**Success criteria:**
- Pre-flight checks all pass
- Workflow spawned with current cache state
- Workflow return handled appropriately

  <action>
Create get-stuff-done/workflows/upstream-sync.md as the 7-stage upstream sync workflow.

**Workflow structure with 7 stages:**

```markdown
# Upstream Sync Workflow

<objective>
Orchestrate the 7-stage upstream sync process: fetch, present, plan, execute, verify, publish, finalize.
</objective>

<stages>

## Stage 1: FETCH
Fetch upstream commits since last sync.

1. Read last_sync.sha from cache (or use initial commit if first sync)
2. Run: `git fetch upstream main`
3. Get new commits: `git log --oneline ${LAST_SHA}..upstream/main`
4. If no new commits: Return `## SYNC COMPLETE` with "Already up to date"
5. Build commit list with summary and files changed per commit

## Stage 2: PRESENT (Checkpoint)
Present commits to user for selection.

Return checkpoint:
```
## CHECKPOINT: CHERRY_PICK_SELECTION

**Commits available since last sync ({last_sha}):**

| SHA | Date | Author | Summary | Files |
|-----|------|--------|---------|-------|
...

**Options:**
1. Enter commit SHAs to cherry-pick (space-separated)
2. Enter "all" to apply all commits
3. Enter "abort" to cancel sync

Awaiting selection...
```

## Stage 3: PLAN
Generate PLAN.md for selected commits.

1. Create `.planning/sync/plans/{date}-{sha}.md` with:
   - Selected commits listed
   - Expected file changes
   - Cherry-pick command sequence
2. Show plan to user for review
3. Continue or abort based on user response

## Stage 4: EXECUTE
Apply cherry-picks.

For each selected commit:
1. Run: `git cherry-pick -x {sha}`
2. If conflict:
   - Run: `git status --short | grep '^UU'`
   - Return `## CONFLICT DETECTED` with file list
   - User resolves manually, runs `git cherry-pick --continue`
   - Workflow resumes
3. After each successful cherry-pick, verify no conflict markers:
   - Run: `git diff --check HEAD~1`

## Stage 5: VERIFY
Run verification on changed files.

1. Get list of changed files from cherry-picks
2. Syntax validation (JSON, YAML, JS files)
3. If tests exist and were passing before: run tests, compare results
4. If new test failures introduced: Return `## VERIFICATION FAILED`
5. Check for conflict markers: `grep -rn '^<<<<<<<' .`
6. Create report at `.planning/sync/reports/{date}-{sha}.md`

## Stage 6: PUBLISH (Checkpoint)
Version bump, changelog, commit, push, publish.

Return checkpoint for version selection:
```
## CHECKPOINT: VERSION_BUMP

**Commits to publish:**
{commit list}

**Recommended version bump:** {major|minor|patch}
Based on: {conventional commit analysis or "manual selection required"}

**Current version:** {from package.json}
**New version:** {calculated}

**Generated changelog entries:**
{Keep A Changelog format entries}

**Registry:** {last used or "public npm"}

**Options:**
1. Accept recommended version
2. Enter custom version (e.g., "1.2.3")
3. Change registry
4. Abort publish

Awaiting selection...
```

After user confirms:
1. Update package.json version
2. Update CHANGELOG.md with generated entries
3. Commit: `git commit -am "chore(release): v{version}"`
4. Push: `git push origin main`
5. Publish with retry logic (3 attempts, exponential backoff)
6. Tag: `git tag v{version} && git push --tags`

## Stage 7: FINALIZE
Update cache and show summary.

1. Update .planning/sync/cache.json:
   - last_sync.sha = latest cherry-picked commit SHA
   - last_sync.date = current timestamp
2. Show completion summary:
```
## SYNC COMPLETE

**Synced:** {N} commits from upstream
**Published:** v{version} to {registry}
**Cache updated:** last_sync.sha = {sha}

Run `/gsd:update` to install the new version.
```

</stages>

<checkpoint_handling>
All checkpoints use AskUserQuestion tool.
Workflow pauses and returns structured checkpoint to orchestrator.
Orchestrator presents to user, collects response.
Orchestrator spawns fresh continuation with user's selection.
</checkpoint_handling>

<error_handling>
- Network errors: Auto-retry with backoff
- Auth errors: Return to orchestrator for user intervention
- Conflict errors: Pause workflow, return CONFLICT DETECTED
- Validation failures: Block publish, show report
</error_handling>
```
