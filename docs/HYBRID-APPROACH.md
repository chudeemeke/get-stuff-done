# Get Stuff Done: Hybrid Approach

**A synthesis of GSD (GlitterCowboy) and Manus.im methodologies, tailored for power users who want the best of both worlds.**

---

## Vision Statement

> "I like taking the best of multiple things and mashing them together as long as I understand them. I don't mind the maintenance required because architecting is part of what I like."

This methodology is designed for developers who:
- Want structured planning without enterprise ceremony
- Need autonomous execution with user checkpoints
- Value clean git history and audit trails
- Appreciate context engineering for quality
- Plan to dogfood (use the tool to build itself)

---

## Core Principles

### Principle 1: Discuss Before Autonomy

**From GSD:** Capture user preferences BEFORE autonomous execution.

> "Most failures come from wrong assumptions."

**Enhancement:** Store discussion artifacts in stable prefix for cache optimization.

```
┌─────────────────────────────────────────────────────────────┐
│                  DISCUSS PHASE                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. GOAL CLARITY                                           │
│      - What exactly are we achieving?                       │
│      - What does "done" look like?                          │
│      - Who is this for?                                     │
│                                                             │
│   2. GRAY AREAS                                             │
│      - What decisions are unmade?                           │
│      - What could go multiple ways?                         │
│      - What are user preferences (not assumptions)?         │
│                                                             │
│   3. CONSTRAINTS                                            │
│      - What must NOT change?                                │
│      - What are the boundaries?                             │
│      - What's explicitly out of scope?                      │
│                                                             │
│   Output: {phase}-CONTEXT.md (stable, cacheable)            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Principle 2: Plan with Backward Reasoning

**From GSD + Enhancement:** Use backward reasoning to derive requirements.

> "What conditions MUST be true for this goal to be achieved?"

This ensures completeness - you can't miss requirements if you reason backward from success.

```markdown
## Backward Reasoning Template

Goal: [Clear statement of success]

### Conditions for Success (What MUST be true?)
1. [Condition A] - Without this, goal fails because...
2. [Condition B] - Without this, goal fails because...
3. [Condition C] - Without this, goal fails because...

### Forward Plan (How to achieve conditions?)
- [ ] Task 1 → achieves Condition A
- [ ] Task 2 → achieves Condition B (part 1)
- [ ] Task 3 → achieves Condition B (part 2)
- [ ] Task 4 → achieves Condition C

### Verification
- [ ] Condition A testable by: [method]
- [ ] Condition B testable by: [method]
- [ ] Condition C testable by: [method]
```

### Principle 3: Fresh Context for Execution

**From GSD:** Each subagent gets isolated, fresh context (200k tokens clean) via the Task tool.

**VERIFIED:** GSD uses the **Task tool** (not TaskCreate/TaskUpdate/TaskList) to spawn subagents with fresh 200k-token contexts. Task state files are auto-generated by Claude Code at `~/.claude/todos/[session-id]-agent-*.json` containing status and activeForm fields.

**11 Custom GSD Agents:**
- gsd-executor (executes plans atomically)
- gsd-planner (creates executable phase plans)
- gsd-phase-researcher (researches implementation approaches)
- gsd-project-researcher (surveys domain ecosystems)
- gsd-verifier (goal-backward verification)
- gsd-plan-checker (validates plans before execution)
- gsd-debugger (scientific method investigation)
- gsd-roadmapper (phase breakdown from requirements)
- gsd-codebase-mapper (brownfield analysis)
- gsd-integration-checker (cross-phase E2E flows)
- gsd-research-synthesizer (consolidates parallel research)

**Enhancement:** Use stable prefix (cached) + fresh suffix (task-specific).

```
┌─────────────────────────────────────────────────────────────┐
│                  CONTEXT STRUCTURE                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   STABLE PREFIX (Cached - rarely changes)                   │
│   ├── System instructions                                   │
│   ├── Project context (PROJECT.md)                          │
│   ├── User preferences (from Discuss phase)                 │
│   └── Domain knowledge                                      │
│                                                             │
│   ─────────────────────────────────────────                 │
│                                                             │
│   FRESH SUFFIX (Per-subagent - spawned via Task tool)       │
│   ├── Current task plan                                     │
│   ├── Relevant file contents                                │
│   ├── Error history (if any)                                │
│   └── Progress state                                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Principle 4: File System as Memory

**From Manus:** Treat persistent storage as unlimited context.

**VERIFIED:** Industry-standard practice confirmed in GSD implementation and multiple open-source agent frameworks (AutoGPT, BabyAGI).

**GSD State File Formats:**
- `STATE.md` - Decisions, blockers, cross-session state
- `PLAN.md` - YAML frontmatter + XML tasks (2-3 tasks max per plan)
- `ROADMAP.md` - Phase definitions with success criteria
- `SUMMARY.md` - What was built, deviations, verification
- `VERIFICATION.md` - Gap analysis (truths, artifacts, key links)
- `.planning/config.json` - Model profiles, execution modes

**Implementation:**
- `todo.md` - Live checklist, updated per step (attention manipulation)
- `scratch/` - Intermediate results, drafts, experiments
- `archive/` - Completed artifacts for reference

### Principle 5: Preserve Error Evidence

**From Manus:** Keep failures visible in context.

**VERIFICATION STATUS:** PARTIAL (documented principle, logical with append-only architecture, but no implementation code or cost-benefit analysis)

**Manus Claim:** Preserve failed actions and error traces in context to allow models to learn from mistakes. Counter-intuitive principle that differs from typical prompt engineering (which emphasizes removing clutter).

**Tension:** Preserving all errors increases token usage and costs, potentially conflicting with KV-cache optimization goals.

**Enhancement:** Also preserve in commit messages and STATE.md.

```markdown
## Error Log (STATE.md section)

### 2026-01-23: Task X Failed
- **Error:** [description]
- **Cause:** [analysis]
- **Resolution:** [what fixed it]
- **Learning:** [what to do differently]
```

### Principle 6: Atomic Commits with Rich Context

**From GSD:** One task = one commit = one thing.

**Enhancement:** Include error history and decision rationale in commit body.

```
feat(auth): implement JWT token generation

Conditions achieved:
- Tokens expire after 1 hour
- Refresh mechanism works
- Secure secret storage

Errors encountered and resolved:
- Initial: bcrypt timing attack vulnerability
- Fix: Switched to constant-time comparison

Decision rationale:
- Chose RS256 over HS256 for key rotation support
```

### Principle 7: Explicit Verification

**From GSD:** User acceptance testing before completion.

**Enhancement:** Continuous observation (Manus) + formal UAT (GSD).

```
┌─────────────────────────────────────────────────────────────┐
│                  VERIFICATION LAYERS                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   LAYER 1: Observation (Continuous)                         │
│   - Check result after each action                          │
│   - Preserve errors if observed                             │
│   - Self-correct before proceeding                          │
│                                                             │
│   LAYER 2: Task Verification (Per Task)                     │
│   - Does output match plan?                                 │
│   - Are acceptance criteria met?                            │
│   - Is commit atomic and reversible?                        │
│                                                             │
│   LAYER 3: UAT (Per Phase)                                  │
│   - User reviews deliverables                               │
│   - Tests against success conditions                        │
│   - Approves or generates fix plans                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## The Four-Phase Cycle (Enhanced)

### Phase 1: DISCUSS (User-Centric)

**Purpose:** Capture implementation preferences BEFORE any planning.

**Key Questions:**
1. Goal clarity - What exactly, for whom, why?
2. Gray areas - What decisions need making?
3. Constraints - What's out of scope?
4. Preferences - How should ambiguity resolve?

**Output:** `{phase}-CONTEXT.md`

**Cache Strategy:** Store in stable prefix (reusable across tasks).

### Phase 2: PLAN (Backward + Forward)

**Purpose:** Research and create atomic task plans.

**Process:**
1. **Backward reasoning:** What must be true for success?
2. **Research:** What patterns exist? What pitfalls?
3. **Forward planning:** Tasks to achieve each condition
4. **Atomicity check:** Each task = one commit?

**Output:** `{phase}-RESEARCH.md`, `{phase}-{N}-PLAN.md`

**Template:**
```markdown
# Plan: {phase}-{N}

## Success Condition
This plan achieves: [specific condition from backward reasoning]

## Acceptance Criteria
- [ ] Criterion 1
- [ ] Criterion 2

## Atomic Tasks
1. [Task with single responsibility]
2. [Task with single responsibility]

## Error Anticipation
- If [X] fails: [contingency]
- If [Y] fails: [contingency]

## Verification Method
[How to test this plan succeeded]
```

### Phase 3: EXECUTE (Autonomous + Observable)

**Purpose:** Run plans with fresh contexts.

**Process:**
1. Load stable prefix (cached context)
2. Load fresh suffix (task plan)
3. Execute with observation loop
4. Commit atomically
5. Update todo.md
6. Proceed to next task

**Rules:**
- One action → observe → next action
- Preserve all errors in context
- Update todo.md after each step (attention manipulation)
- Commit immediately after task completion

**Output:** `{phase}-{N}-SUMMARY.md`

### Phase 4: VERIFY (Formal UAT)

**Purpose:** User acceptance testing.

**Process:**
1. Extract testable deliverables
2. Walk through each with user
3. Test against success conditions
4. Generate fix plans if failures
5. Loop until acceptance

**Output:** `{phase}-VERIFICATION.md`, `{phase}-UAT.md`

## Integration with WoW System

**Consideration:** How does Get Stuff Done interact with WoW (Ways of Working) defensive security?

### Potential Synergies

| GSD Phase | WoW Integration |
|-----------|-----------------|
| **Discuss** | WoW could enforce that Discuss phase happens before execution |
| **Plan** | WoW could validate plans against security zones before execution |
| **Execute** | WoW's handler-router intercepts tool calls during execution |
| **Verify** | WoW's scoring could factor into verification criteria |

### Potential Tensions

| Tension | Resolution |
|---------|------------|
| GSD wants autonomous execution | WoW blocks dangerous operations | Use `wow bypass` for trusted autonomous phases |
| GSD uses fresh contexts | WoW state persists across sessions | WoW state files are outside GSD context |
| GSD commits atomically | WoW might block git operations | Ensure git operations are in allowed zones |

### Design Decision (Deferred)

**Question:** Should GSD be aware of WoW, or should they be orthogonal?

**Options:**
1. **Orthogonal:** GSD manages tasks, WoW manages security - no direct integration
2. **Integrated:** GSD checks WoW status before execution, handles bypasses
3. **Layered:** WoW wraps GSD (GSD runs inside WoW's protection)

**Current Recommendation:** Start orthogonal, integrate later based on friction points.

## Dogfooding Strategy

**Goal:** Use Get Stuff Done to build Get Stuff Done.

### Bootstrap Phases

1. **Phase 0:** Manual GSD-Lite to build core infrastructure
2. **Phase 1:** Use infrastructure to build command system
3. **Phase 2:** Use commands to build advanced features
4. **Phase 3:** Full self-hosting

### Quality Signal

The methodology is working if:
- Each phase produces cleaner code than manual work
- Context rot is not observed
- Commits are atomic and reversible
- User preferences are respected
- Errors are learned from (not repeated)

### Command Reference (Future)

```bash
# Project initialization
gsd init                    # Create project structure

# Planning
gsd discuss                 # Start Discuss phase
gsd plan                    # Create plans from context
gsd plan --quick            # Skip research, minimal planning

# Execution
gsd execute                 # Run all pending plans
gsd execute --plan 1        # Run specific plan

# Verification
gsd verify                  # Start UAT
gsd verify --auto           # Auto-verify (tests only)

gsd status                  # Show current state
gsd state                   # Edit STATE.md
gsd todo                    # Edit todo.md
```

## Manus Principles: Feasibility Assessment

### ✅ HIGH-VALUE PRINCIPLES (Adopt in Phase 1)

**1. Design Around KV-Cache**
- **Status:** VERIFIED (documented with technical detail, independent validation exists)
- **Feasibility:** PARTIAL - Can implement stable prefixes, append-only context, deterministic JSON
- **Cannot Replicate:** Explicit cache breakpoints (requires API support), measuring actual cache hit rate
- **Expected Benefit:** 10x cost reduction ($0.30 vs $3 per MTok with Claude Sonnet)
- **Implementation Timeline:** 2-3 days

**2. File System as Context**
- **Status:** VERIFIED (industry standard, confirmed in GSD + community replications)
- **Feasibility:** 100%
- **Implementation:** Workspace per session, save tool outputs to files, RAG for retrieval
- **Expected Benefit:** Significant token savings, persistence across sessions
- **Implementation Timeline:** 1-2 days

**3. Keep the Wrong Stuff In**
- **Status:** PARTIAL (documented, no implementation code or cost-benefit analysis)
- **Feasibility:** 90% - Architecturally simple
- **Implementation:** Append-only event log, preserve errors, include traces in context
- **Risk:** Context window fills faster, costs increase
- **Implementation Timeline:** Built-in if using append-only architecture

### ⚠️ MEDIUM-VALUE PRINCIPLES (Consider for Phase 2)

**4. Mask, Don't Remove**
- **Status:** PARTIAL (well-documented principle, no implementation evidence)
- **Feasibility:** 30% - Requires low-level model access
- **Blocker:** Logits masking not available via Claude API
- **Workaround:** Constrain via system prompt ("only use these tools"), post-validation
- **Expected Benefit:** Cleaner than dynamic tool removal
- **Implementation Timeline:** N/A unless self-hosted model

### ❌ LOW-CONFIDENCE PRINCIPLES (Skip or Defer)

**5. Manipulate Attention Through Recitation**
- **Status:** PARTIAL (documented but vague, no implementation evidence)
- **Feasibility:** 60% - Implementable but unclear value
- **Missing:** How often to recite? What triggers recitation? Does it help performance?
- **Risk:** Token waste if done too frequently
- **Recommendation:** SKIP until evidence proves value

**6. Don't Get Few-Shotted**
- **Status:** PARTIAL (documented, no implementation guidance)
- **Feasibility:** 40% - Conflicts with KV-cache optimization
- **Critical Tension:** Requires variation (Principle 6) while demanding stability (Principle 1)
- **Missing:** How to balance variation without breaking cache? What is "controlled" variation?
- **Recommendation:** SKIP - appears contradictory to Principle 1
