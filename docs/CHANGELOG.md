### Research Phase (2026-01-24 to 2026-01-25)

#### Phase 1: GSD Deep Dive Research (2026-01-25)

**Research Output:** `research/01-gsd-deep-dive.md` (1537 lines)

**Objectives:**
- Extract complete understanding of GSD repository architecture
- Document all custom agent definitions with frontmatter
- Analyze orchestration patterns and state file formats
- Verify Task tool usage (NOT TaskCreate/TaskUpdate for todo tracking)
- Document model selection system

**Findings:**

##### Custom Agents Documented (11 total)
1. **gsd-executor** - Executes plans atomically with deviation handling, checkpoints
2. **gsd-planner** - Creates executable phase plans with backward reasoning
3. **gsd-phase-researcher** - Researches implementation approaches before planning
4. **gsd-project-researcher** - Surveys domain ecosystems for roadmap creation
5. **gsd-verifier** - Goal-backward verification of code vs requirements
6. **gsd-plan-checker** - Validates plans before execution
7. **gsd-debugger** - Scientific method investigation with hypothesis testing
8. **gsd-roadmapper** - Transforms requirements into structured phases
9. **gsd-codebase-mapper** - Analyzes brownfield codebases across 4 dimensions
10. **gsd-integration-checker** - Verifies cross-phase E2E flows
11. **gsd-research-synthesizer** - Consolidates parallel research streams

##### Orchestration Patterns (6 identified)
- Sequential-with-Parallel Phases (research parallelization)
- Wave-Based Parallel Execution (dependency-aware task waves)
- Iterative Verification Loop (plan validation with revision)
- Goal-Backward Validation (3-level artifact verification)
- Checkpoint-Based Pause-Resume (90% human-verify, 9% decision, 1% action)
- Context-Preserving Orchestration (30-40% main utilization)

##### State File Formats (8 documented)
- `PLAN.md` - YAML frontmatter + XML tasks (2-3 tasks max per plan)
- `STATE.md` - Decisions, blockers, quick tasks tracker
- `ROADMAP.md` - Phase definitions with success criteria
- `SUMMARY.md` - Post-execution deliverables and deviations
- `VERIFICATION.md` - Gap analysis with YAML frontmatter
- `RESEARCH.md` - Phase research with confidence levels
- `.planning/config.json` - Model profiles and execution modes
- Task state files (auto-generated by Claude Code at `~/.claude/todos/`)

##### Model Selection System
- 4 profiles: Quality (all Opus), Balanced (all Sonnet), Budget (Haiku/Sonnet mix), Adaptive (task-complexity-based)
- Adaptive selection: Simple (Haiku) → Medium (Sonnet) → Complex (Opus)
- Cost savings: ~62% vs quality profile, ~35% vs balanced
- Per-agent model override support

##### Task Tool Clarification
**VERIFIED:** GSD uses Task tool for spawning subagents with fresh 200k contexts, NOT for traditional todo tracking.
- Each Task invocation creates isolated subagent context
- Prevents context rot through context isolation
- Enables parallel execution of independent work streams
- State files auto-generated by Claude Code, not manually via TaskCreate API

##### Error Handling Patterns (12 categories)
- Blocking scenarios with graceful recovery
- Graceful fallbacks for missing configuration
- Checkpoint-based error recovery
- Deviation handling (4 automatic rules)
- Verification failure handling
- Plan verification failure handling
- Context window management
- File system error handling
- Rate limit handling (adaptive profile)
- Git commit failure handling
- Debug session management

#### Phase 2: Manus Verification Research (2026-01-25)

**Research Output:** `research/02-manus-verification.md` (650+ lines)

**Objectives:**
- Verify 6 context engineering principles against public documentation
- Assess implementation feasibility for each principle
- Identify open-source replications and evaluate code availability
- Document critical tensions between principles

**Findings:**

##### Principle Verification Status

| Principle | Documentation | Code Evidence | Replication | Status |
|-----------|---------------|---------------|-------------|--------|
| 1. KV-Cache Optimization | ✅ Detailed | ❌ None | ⚠️ Partial | **VERIFIED** (principle) |
| 2. Mask, Don't Remove | ✅ Described | ❌ None | ❌ None | **PARTIAL** |
| 3. File System as Context | ✅ Detailed | ✅ Gist | ✅ Yes | **VERIFIED** |
| 4. Attention Recitation | ⚠️ Vague | ❌ None | ❌ None | **PARTIAL** |
| 5. Keep Wrong Stuff In | ✅ Described | ❌ None | ❌ None | **PARTIAL** |
| 6. Don't Get Few-Shotted | ⚠️ Vague | ❌ None | ❌ None | **PARTIAL** |

##### Critical Gap: No Open-Source Code
- Manus acquired by Meta for >$2 billion (December 2025)
- Core implementation remains proprietary trade secret
- Blog posts are directional guidance, not blueprints
- Community must reverse-engineer from documentation

##### Open-Source Implementations Found
1. **OpenManus (FoundationAgents)** - Built in 3 hours by MetaGPT team, missing advanced features
2. **OpenManus (henryalps)** - Modular framework, no context engineering
3. **AI Manus (Simpleyyt)** - Task isolation and monitoring, no KV-cache optimization
4. **manus-open (whit3rabbit)** - Reconstructed from container bytecode (unverified)

**Key Insight:** Basic agent loop is simple to replicate, but advanced context engineering features (KV-cache, logits masking, recitation) are proprietary/missing.

##### Critical Design Tension Identified
**Principle 1 (KV-Cache)** demands stable prompts vs **Principle 6 (Don't Get Few-Shotted)** demands variation.

**Status:** UNVERIFIED - No public code shows resolution strategy.

##### Implementation Feasibility Assessment

**HIGH-VALUE PRINCIPLES (Adopt Phase 1):**
1. **KV-Cache Optimization** - Feasible: stable prefixes, append-only, deterministic JSON. Cannot replicate: explicit cache breakpoints, measuring hit rate. Expected benefit: 10x cost reduction.
2. **File System as Context** - 100% feasible, industry standard, 1-2 day implementation.
3. **Keep Wrong Stuff In** - 90% feasible, built-in with append-only architecture.

**MEDIUM-VALUE PRINCIPLES (Phase 2):**
4. **Mask, Don't Remove** - 30% feasible (requires logits masking API). Workaround: prompt-based constraints.

**LOW-CONFIDENCE PRINCIPLES (Skip/Defer):**
5. **Attention Recitation** - 60% implementable, unclear value, no benchmarks.
6. **Don't Get Few-Shotted** - 40% feasible, conflicts with KV-cache, appears contradictory.

##### API Surface Documented

**TaskCreate:**
- subject (string, required)
- description (string, required)
- activeForm (string, optional) - spinner text when in_progress
- metadata (object, optional)

**TaskUpdate:**
- taskId (string, required)
- status (pending | in_progress | completed)
- addBlockedBy (array) - tasks that block this
- addBlocks (array) - tasks this blocks
- metadata (object) - merge keys

**TaskList:**
- No parameters, returns all tasks

**TaskGet:**
- taskId (string, required)
- Returns full task object with relationships

**Task:**
- Spawns subagent in fresh 200k context
- agent, description, prompt, model parameters

##### Key Discoveries
1. **Auto-unblocking** - Dependencies resolve automatically when blocking task completes
2. **activeForm field** - Provides user-friendly progress text
3. **Metadata support** - Arbitrary tracking data
4. **Sequential task IDs** - Simple numeric (#1, #2, etc.)
5. **Session-scoped** - Tasks don't persist across restarts (need STATE.md sync)

##### Custom Agent Testing
- Created `~/.gsd/agents/test-agent.md` with YAML frontmatter
- File structure verified, invocation not tested in session

##### Parallel Execution (Documented)
✅ Multiple Task tool invocations can run in parallel (wave pattern)
- Independent tasks spawn simultaneously
- Each gets fresh 200k context
- Dependency boundaries enforced by waves

### Decisions Made

#### DR-001: Hybrid Approach (Accepted)
- **Date:** 2026-01-23
- **Decision:** Combine GSD + Manus rather than adopt either wholesale
- **Rationale:** User values understanding and custom solutions, willing to maintain
- **Status:** Accepted
- **Documentation:** `decisions/001-HYBRID-RATIONALE.md`

#### DR-002: Option A First, Option C Later (Accepted)
- **Date:** 2026-01-24
- **Decision:** Build Claude Code commands (Option A) with structure enabling CLI pivot (Option C)
- **Key Requirements:** YAML frontmatter, JSON schemas, methodology/implementation separation
- **Pivot Cost Estimate:** ~2 weeks vs 2+ months starting fresh
- **Documentation:** `architecture/SYSTEM-DESIGN.md` (Option C section)

#### DR-003: Clone GSD Agent Approach (Accepted)
- **Date:** 2026-01-24
- **Decision:** Use GSD's custom agent architecture enhanced with Claude Code 2.1.16+ features
- **Enhancement:** Integrate TaskCreate/TaskUpdate/TaskList for progress tracking
- **Rationale:** Battle-tested patterns + modern capabilities
- **Documentation:** `architecture/CLAUDE-CODE-INTEGRATION.md`

#### DR-004: Research-First Quality Gate (Accepted)
- **Date:** 2026-01-24
- **Decision:** Pause WoW v8.0 work until Get Stuff Done methodology verified
- **Phases:** 5 research/verification phases before dogfooding
- **Rationale:** Prevent "using incomplete GSD to build WoW" quality issues
- **Status:** Phases 1-3 complete (2026-01-25)
