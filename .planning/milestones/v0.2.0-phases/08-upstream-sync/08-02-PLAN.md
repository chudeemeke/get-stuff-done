**IMPORTANT: This task is an iterative loop, not a single action. It processes ALL pending commits from the sync manifest. If context fills before completion, the manifest state enables the next session to resume.**

**Session Resume Protocol:**
Before starting, read `.planning/sync/sync-manifest.json`. If `status` is "in_progress" and there are already applied commits, skip those and continue from the first pending commit. If `status` is "conflict", the previous session hit a conflict -- check if it was resolved (no conflict markers in working tree) and continue.

**Initialize conflict log** (if it does not already exist):

**For EACH pending commit in the manifest (in order):**

1. **Read commit info:**
   ```bash
   git show <sha> --stat --format="%H %s%n%b"
   ```

2. **Show diff preview to user** (mandatory -- SEC-01 requirement):
   ```bash
   git show <sha> --color=always
   ```
   Present the diff statistics and key changes. For large diffs (>200 lines), summarize the changes and show the stat output.

3. **Determine if trivial:** Version bump commits (message matches `^[0-9]+\.[0-9]+`), pure changelog/README docs commits can be noted as trivial. Per context decision, trivial commits MAY be batched -- present 2-5 trivial commits together for approval if consecutive.

4. **Create snapshot tag BEFORE cherry-pick:**
   ```bash
   git tag sync-snapshot-<short-sha>
   ```

5. **Execute cherry-pick:**
   ```bash
   git cherry-pick -x <full-sha>
   ```

6. **If cherry-pick succeeds (exit 0):**
   a. Run ESLint validation: `bunx eslint .`
   b. If ESLint fails: fix the violations (typically security plugin warnings on new code), amend the cherry-pick commit: `git add -A && git commit --amend --no-edit --author="Chude <chude@emeke.org>"`
   c. Update sync-manifest.json: move commit from `pending` to `applied` with timestamp
   d. Update stats

7. **If cherry-pick conflicts:**
   a. Identify conflicted files: `git diff --name-only --diff-filter=U`
   b. Log to conflict-log.md with file, commit SHA, and conflict type
   c. Apply resolution policy:
      - **Protected paths (branding, Phase 7 security):** Fork wins. Use `git checkout --ours <file>` for fully protected files.
      - **get-shit-done/ vs get-stuff-done/ path conflicts:** These are NOT real conflicts -- the fork renamed the directory. Accept upstream content but place it in `get-stuff-done/` path. The final branding pass (Plan 03) handles content renaming.
      - **All other files:** Upstream wins. Use `git checkout --theirs <file>` for clean upstream wins. For mixed conflicts (upstream changes + fork changes both matter), manually merge keeping upstream changes and re-applying fork-specific parts.
      - **package.json:** Special handling -- accept new scripts and dependencies from upstream, but ALWAYS preserve fork's name, version, author, repository URL, homepage, and description fields.
   d. After resolution: `git add <resolved-files> && git cherry-pick --continue`
   e. Verify no conflict markers remain: `git diff --check`
   f. Run ESLint validation
   g. Update conflict-log.md with resolution chosen and reasoning
   h. Update sync-manifest.json: move to applied, record conflict details

8. **After each commit (or batch), save manifest** using atomic write (write to .tmp, rename).

9. **Periodically commit manifest updates** (every 5-10 cherry-picks):
   ```bash
   git add .planning/sync/sync-manifest.json .planning/phases/08-upstream-sync/conflict-log.md
   git commit --author="Chude <chude@emeke.org>" -m "chore(08-sync): update sync progress (N/72 commits applied)"
   ```
