**Objective section:**
Sync commits from glittercowboy/get-shit-done, allow selection/cherry-pick, verify changes, and publish to npm with proper versioning.

**Process section - Pre-flight checks:**
1. Verify git working directory is clean (no uncommitted changes)
   - If dirty: ERROR with `git status` output, require user to commit or stash
2. Verify GitHub auth (SSH key or gh CLI)
   - Run: `git ls-remote --exit-code git@github.com:chudeemeke/get-stuff-done.git HEAD`
   - If fails: Offer to run `gh auth login`
3. Verify npm auth
   - Run: `bun pm whoami` or `npm whoami`
   - If fails: Offer to run `npm login`
4. Add upstream remote if missing
   - Check: `git config --get remote.upstream.url`
   - If missing: `git remote add upstream https://github.com/glittercowboy/get-shit-done.git`

**Process section - Spawn workflow (initial):**
After pre-flight passes, spawn the upstream-sync workflow:

```markdown

  subagent_type="general-purpose",
  model="opus",
  description="Upstream sync workflow - Stage 1"
)
```

**Process section - Handle workflow returns:**

The workflow returns control by outputting a structured section to stdout. The orchestrator (this skill) parses the output and handles each return type:

1. **`## SYNC COMPLETE`** - Display summary, update cache, show success
   - No continuation needed - workflow finished

2. **`## CHECKPOINT: CHERRY_PICK_SELECTION`** - Commit selection checkpoint
   - Present the checkpoint content to user via AskUserQuestion
   - Collect user response (commit SHAs, "all", or "abort")
   - Spawn continuation:
   ```markdown
   Task(
     prompt="First, read ~/.claude/get-stuff-done/workflows/upstream-sync.md.\n\n<sync_context>\nresume_stage: 3\nuser_selection: {user_response}\ncommit_list: {commits_from_checkpoint}\n{cache_json_content}\n</sync_context>",
     subagent_type="general-purpose",
     model="opus",
     description="Upstream sync workflow - Stage 3 (after commit selection)"
   )
   ```

3. **`## CHECKPOINT: VERSION_BUMP`** - Version selection checkpoint
   - Present version options and generated changelog to user
   - Collect user response (accept, custom version, or abort)
   - Spawn continuation:
   ```markdown
   Task(
     prompt="First, read ~/.claude/get-stuff-done/workflows/upstream-sync.md.\n\n<sync_context>\nresume_stage: 6b\nversion_selection: {user_response}\nchangelog_entries: {entries_from_checkpoint}\n{cache_json_content}\n</sync_context>",
     subagent_type="general-purpose",
     model="opus",
     description="Upstream sync workflow - Stage 6b (after version selection)"
   )
   ```

4. **`## SYNC ABORTED`** - Show reason, no changes made
   - No continuation needed - user chose to abort

5. **`## CONFLICT DETECTED`** - Show conflicted files, wait for manual resolution
   - Display conflicted file list
   - Instruct user to resolve conflicts and run `git cherry-pick --continue`
   - After user confirms resolution, spawn continuation at Stage 4

Create get-stuff-done/workflows/upstream-sync.md with stages 1-4 (fetch through execute).

**File structure:**

```markdown
# Upstream Sync Workflow

<objective>
Orchestrate the 7-stage upstream sync process: fetch, present, plan, execute, verify, publish, finalize.

This workflow is spawned by /gsd:upstream with a sync_context containing:
- resume_stage: Which stage to start at (1 for fresh, 3/4/6b for continuations)
- user_selection: User's commit selection (for Stage 3 continuation)
- version_selection: User's version choice (for Stage 6b continuation)
- cache.json content for last sync SHA
</objective>

<checkpoint_protocol>
**Return mechanism:** Output checkpoint section to stdout. Orchestrator parses output and handles.

**Format:**
```
## CHECKPOINT: {CHECKPOINT_NAME}

{structured content for user}

**Options:**
{numbered options}

Awaiting selection...
```

**Continuation:** Orchestrator spawns new Task with resume_stage and user response in sync_context.
</checkpoint_protocol>

<stages>

## Stage 1: FETCH
Fetch upstream commits since last sync.

**Entry:** resume_stage = 1 (or missing)

**Process:**
1. Read last_sync.sha from sync_context (or use initial commit if null/first sync)
2. Run: `git fetch upstream main`
3. Get new commits:
   ```bash
   git log --oneline --format="%H|%ad|%an|%s" --date=short ${LAST_SHA}..upstream/main
   ```
4. If no new commits: Return `## SYNC COMPLETE` with "Already up to date"
5. For each commit, get files changed:
   ```bash
   git diff-tree --no-commit-id --name-status -r {SHA}
   ```
6. Build commit list with: SHA, date, author, summary, files changed

**Output:** Continue to Stage 2 with commit_list

## Stage 2: PRESENT (Checkpoint)
Present commits to user for selection.

**Entry:** After Stage 1 completes

**Process:**
1. Format commit list as table
2. Output checkpoint to stdout (orchestrator will parse):

```
## CHECKPOINT: CHERRY_PICK_SELECTION

**Commits available since last sync ({last_sha}):**

| SHA (short) | Date | Author | Summary | Files Changed |
|-------------|------|--------|---------|---------------|
| abc1234 | 2024-01-15 | glittercowboy | fix: typo in help text | 2 |
| def5678 | 2024-01-14 | glittercowboy | feat: add new command | 5 |
...

**Options:**
1. Enter commit SHAs to cherry-pick (space-separated, e.g., "abc1234 def5678")
2. Enter "all" to apply all commits in order
3. Enter "abort" to cancel sync

Awaiting selection...
```

**Output:** Workflow pauses. Orchestrator presents to user, spawns continuation at Stage 3 with user_selection.

## Stage 3: PLAN
Generate PLAN.md for selected commits.

**Entry:** resume_stage = 3, user_selection in sync_context

**Process:**
1. Parse user_selection:
   - "abort" -> Return `## SYNC ABORTED` with reason "User cancelled"
   - "all" -> Use full commit_list from Stage 1
   - SHAs -> Filter commit_list to selected commits only
2. Create plan file at `.planning/sync/plans/{YYYY-MM-DD}-{first_sha_short}.md`:
   ```markdown
   # Upstream Sync Plan

   **Date:** {timestamp}
   **Commits:** {N} selected

   ## Commits to Apply

   | Order | SHA | Summary |
   |-------|-----|---------|
   | 1 | {sha} | {summary} |
   ...

   ## Expected File Changes

   {aggregated list of files from all selected commits}

   ## Cherry-pick Sequence

   ```bash
   git cherry-pick -x {sha1}
   git cherry-pick -x {sha2}
   ...
   ```
   ```
3. Display plan summary to user (via console output, not checkpoint)
4. Continue to Stage 4

**Output:** Continue to Stage 4 with selected_commits

## Stage 4: EXECUTE
Apply cherry-picks.

**Entry:** After Stage 3, or resume_stage = 4 (after conflict resolution)

**Process:**
For each selected commit in order:
1. Run: `git cherry-pick -x {sha}`
2. Check exit code:
   - Success (0): Continue to next commit
   - Conflict (1): Handle conflict (see below)
3. After each successful cherry-pick, verify no conflict markers:
   ```bash
   git diff --check HEAD~1
   ```

**Conflict handling:**
1. Run: `git status --short | grep '^UU\|^AA\|^DD'`
2. Return:
   ```
   ## CONFLICT DETECTED

   **Conflicted files:**
   {file list}

   **Instructions:**
   1. Open the conflicted files and resolve the conflicts
   2. Stage resolved files: `git add {file}`
   3. Continue cherry-pick: `git cherry-pick --continue`
   4. Return here and type "resolved" to continue sync

   Or type "abort" to cancel (will run `git cherry-pick --abort`)
   ```
3. Workflow pauses. After user resolves:
   - "resolved" -> Continue with remaining commits
   - "abort" -> Run `git cherry-pick --abort`, return `## SYNC ABORTED`

**Output:** After all commits applied, continue to Stage 5

</stages>
```

**Add to the `<stages>` section:**

```markdown
## Stage 5: VERIFY
Run verification on changed files.

**Entry:** After Stage 4 completes successfully

**Process:**
1. Get list of changed files from cherry-picks:
   ```bash
   git diff --name-only HEAD~{N}..HEAD
   ```
   (where N = number of cherry-picked commits)

2. Syntax validation by file type:
   - JSON: `python3 -m json.tool {file} > /dev/null 2>&1`
   - YAML: `python3 -c "import yaml; yaml.safe_load(open('{file}'))" 2>&1`
   - JS/TS: `node --check {file} 2>&1` (syntax only)
   - MD: Skip (no syntax check needed)

3. If tests exist and were passing before:
   ```bash
   bun test 2>&1 || npm test 2>&1
   ```
   Compare exit code to pre-sync state.

4. Check for conflict markers:
   ```bash
   grep -rn '^<<<<<<<\|^=======\|^>>>>>>>' . --include='*.md' --include='*.js' --include='*.ts' --include='*.json'
   ```
   If found: Return `## VERIFICATION FAILED` with file list

5. Create report at `.planning/sync/reports/{YYYY-MM-DD}-{sha}.md`:
   ```markdown
   # Sync Verification Report

   **Date:** {timestamp}
   **Commits:** {N}

   ## Validation Results

   | File | Type | Status |
   |------|------|--------|
   | path/to/file.json | JSON | PASS |
   ...

   ## Test Results

   {test output or "No tests configured"}

   ## Skipped Files

   {files with unknown types}
   ```

6. If any validation failures: Return `## VERIFICATION FAILED` with report path

**Output:** If all pass, continue to Stage 6

## Stage 6: PUBLISH (Checkpoint)
Version bump, changelog, commit, push, publish.

**Entry:** After Stage 5 completes, or resume_stage = 6b (after version selection)

**Stage 6a - Version analysis and checkpoint (if resume_stage != 6b):**

1. Analyze commits to suggest version bump:
   ```bash
   # Check for BREAKING in commit messages
   git log --oneline HEAD~{N}..HEAD | grep -i "BREAKING\|!" && echo "major"
   # Check for feat: prefix
   git log --oneline HEAD~{N}..HEAD | grep -i "^feat\|feat:" && echo "minor"
   # Default to patch
   ```

2. Get current version:
   ```bash
   cat package.json | grep '"version"' | sed 's/.*: "\(.*\)".*/\1/'
   ```

3. Calculate new version based on bump type

4. Generate changelog entries (Keep A Changelog format):
   ```markdown
   ## [{new_version}] - {YYYY-MM-DD}

   ### Added
   - {feat commits}

   ### Changed
   - {refactor/chore commits}

   ### Fixed
   - {fix commits}
   ```

5. Return checkpoint:
   ```
   ## CHECKPOINT: VERSION_BUMP

   **Commits to publish:** {N}

   **Recommended version bump:** {major|minor|patch}
   Based on: {commit analysis or "manual selection required"}

   **Current version:** {from package.json}
   **New version:** {calculated}

   **Generated changelog entries:**
   {Keep A Changelog format entries}

   **Registry:** {last used from cache or "public npm"}

   **Options:**
   1. Accept recommended version (enter "accept")
   2. Enter custom version (e.g., "1.2.3")
   3. Change registry (enter "registry")
   4. Abort publish (enter "abort")

   Awaiting selection...
   ```

**Stage 6b - Execute publish (resume_stage = 6b):**

Parse version_selection from sync_context:
- "abort" -> Return `## SYNC ABORTED` with reason "User cancelled publish"
- "accept" -> Use calculated version
- Custom version -> Validate semver format, use if valid
- "registry" -> (handled by orchestrator, re-prompts)

**Execute publish sequence with specific commands:**

1. **Update package.json version:**
   ```bash
   # Using bun (preferred) or npm
   bun version {new_version} --no-git-tag-version 2>/dev/null || \
   npm version {new_version} --no-git-tag-version
   ```

2. **Update CHANGELOG.md:**
   - Read existing CHANGELOG.md
   - Insert generated entries after "## [Unreleased]" heading (or at top if no Unreleased section)
   - Write updated CHANGELOG.md

3. **Commit changes:**
   ```bash
   git add package.json CHANGELOG.md
   git commit -m "chore(release): v{new_version}

   Synced {N} commits from upstream:
   - {commit summaries truncated to 3}"
   ```

4. **Push to remote:**
   ```bash
   git push origin main
   ```
   On failure: Check error type
   - Network error -> Auto-retry (up to 3 times, 2s delay)
   - Auth error -> Return to orchestrator: "GitHub push failed: authentication required"
   - Rejection -> Return to orchestrator: "Push rejected - remote has changes"

5. **Publish to npm with retry logic:**
   ```bash
   # Attempt 1
   bun publish --access public 2>&1 || npm publish --access public 2>&1
   # If fails, wait 1 second
   sleep 1
   # Attempt 2
   bun publish --access public 2>&1 || npm publish --access public 2>&1
   # If fails, wait 3 seconds
   sleep 3
   # Attempt 3
   bun publish --access public 2>&1 || npm publish --access public 2>&1
   ```

   On all attempts failed:
   ```
   ## PUBLISH FAILED

   **Error:** {last error message}

   **Recovery instructions:**
   1. Check npm auth: `npm whoami`
   2. If auth expired: `npm login`
   3. Retry manually: `bun publish --access public`
   4. If registry issue: Check https://status.npmjs.org

   **State:** Git commit and push succeeded. Version {version} is committed but not published.
   To retry: Run `/gsd:upstream` again - it will detect unpublished version.
   ```

6. **Create and push tag:**
   ```bash
   git tag v{new_version}
   git push origin v{new_version}
   ```

**Output:** If all succeed, continue to Stage 7

## Stage 7: FINALIZE
Update cache and show summary.

**Entry:** After Stage 6 completes successfully

**Process:**
1. Update .planning/sync/cache.json:
   ```bash
   # Read current cache
   CACHE=$(cat .planning/sync/cache.json)

   # Update last_sync fields
   echo "$CACHE" | \
     sed "s/\"sha\": [^,]*/\"sha\": \"{latest_cherry_picked_sha}\"/" | \
     sed "s/\"date\": [^,]*/\"date\": \"$(date -Iseconds)\"/" > \
     .planning/sync/cache.json.tmp && \
     mv .planning/sync/cache.json.tmp .planning/sync/cache.json
   ```

2. Return completion:
   ```
   ## SYNC COMPLETE

   **Synced:** {N} commits from upstream
   **Published:** v{version} to {registry}
   **Cache updated:** last_sync.sha = {sha}

   **Summary:**
   {brief list of changes}

   Run `/gsd:update` to install the new version locally.
   ```

</stages>

<error_handling>
- **Network errors:** Auto-retry with backoff (1s, 3s, 5s)
- **Auth errors:** Return to orchestrator for user intervention (no auto-retry)
- **Conflict errors:** Pause workflow, return CONFLICT DETECTED, resume after resolution
- **Validation failures:** Block publish, show report path
- **Publish failures:** 3 retries with exponential backoff, then recovery instructions
</error_handling>
```
