**hooks/gsd-check-update.js:**

The spawned child process at line 27 currently uses:
```javascript
const { execSync } = require('child_process');
latest = execSync('npm view @chude/get-stuff-done version', { encoding: 'utf8', timeout: 10000, windowsHide: true }).trim();
```

Replace `execSync` with `execFileSync` in the spawned child code (the string passed to `spawn`). This is the inline code string passed to `spawn(process.execPath, ['-e', ...])`:

Change to:
```javascript
const { execFileSync } = require('child_process');
latest = execFileSync('npm', ['view', '@chude/get-stuff-done', 'version'], { encoding: 'utf8', timeout: 10000, windowsHide: true }).trim();
```

Important: On Windows, `execFileSync('npm', ...)` needs the npm executable to be found. Since this runs through Node.js and npm is typically available on PATH, this should work. However, on Windows specifically, npm is actually `npm.cmd`. To handle this cross-platform, use `{ shell: false }` (default for execFileSync) but add a Windows fallback:

Actually, the safest approach for this specific case: since the command is entirely hardcoded (no user input), and we just want to avoid shell interpretation of the npm output, the simplest fix is to keep using execSync but explicitly note it's safe because the command string is a compile-time constant with no interpolation. Add a clear eslint-disable comment. The real risk is commands with user input -- this hardcoded npm view command has no injection vector.

Instead, focus the hardening effort on the actual attack surface:

For gsd-check-update.js: Add an eslint-disable comment on the execSync line explaining why it's acceptable:
```javascript
// eslint-disable-next-line security/detect-child-process -- hardcoded command string, no user input, safe
latest = execSync('npm view @chude/get-stuff-done version', { encoding: 'utf8', timeout: 10000, windowsHide: true }).trim();
```

**bin/gsd (bash script):**

The `read_config` function at line 60-75 uses string interpolation to inject variables into `node -e`:
```bash
node -e "
    const {loadConfig, getConfigValue} = require('${GSD_INSTALL_DIR}/src/config/ConfigLoader');
    try {
        const config = loadConfig();
        const val = getConfigValue(config, '${key}', '${default}');
        console.log(val);
    } catch (e) {
        console.error('Config error:', e.message);
        console.log('${default}');
    }
" 2>/dev/null || echo "$default"
```

This interpolates `GSD_INSTALL_DIR`, `key`, and `default` directly into JavaScript code. While these come from the script itself (not external user input), this is a command injection pattern. Harden by passing values as environment variables instead:

Replace the `read_config` function with:
```bash
read_config() {
    local key="$1"
    local default="$2"

    GSD_INSTALL_DIR="$GSD_INSTALL_DIR" \
    GSD_CONFIG_KEY="$key" \
    GSD_CONFIG_DEFAULT="$default" \
    node -e '
        const {loadConfig, getConfigValue} = require(process.env.GSD_INSTALL_DIR + "/src/config/ConfigLoader");
        try {
            const config = loadConfig();
            const val = getConfigValue(config, process.env.GSD_CONFIG_KEY, process.env.GSD_CONFIG_DEFAULT);
            console.log(val);
        } catch (e) {
            console.error("Config error:", e.message);
            console.log(process.env.GSD_CONFIG_DEFAULT);
        }
    ' 2>/dev/null || echo "$default"
}
```

Key changes:
- Single-quoted node -e string (no bash interpolation)
- Values passed via environment variables (GSD_CONFIG_KEY, GSD_CONFIG_DEFAULT)
- GSD_INSTALL_DIR also passed via env (already set, but re-exported for clarity)
- process.env used inside JS to read values

Apply the same pattern to the `ensure_config` function (lines 115-138) which also uses `node -e` with string interpolation of `GSD_INSTALL_DIR` and `GSD_CONFIG`:

Replace the two `node -e` blocks in ensure_config with the same env-var pattern:
```bash
# For has_version check:
has_version=$(GSD_INSTALL_DIR="$GSD_INSTALL_DIR" GSD_CONFIG="$GSD_CONFIG" \
    node -e '
        const JSON5 = require(process.env.GSD_INSTALL_DIR + "/node_modules/json5");
        const fs = require("fs");
        try {
            const config = JSON5.parse(fs.readFileSync(process.env.GSD_CONFIG, "utf8"));
            console.log(config.version ? "yes" : "no");
        } catch (e) {
            console.log("no");
        }
    ' 2>/dev/null || echo "no")

# For migration:
GSD_INSTALL_DIR="$GSD_INSTALL_DIR" GSD_CONFIG="$GSD_CONFIG" \
    node -e '
        const JSON5 = require(process.env.GSD_INSTALL_DIR + "/node_modules/json5");
        const fs = require("fs");
        const config = JSON5.parse(fs.readFileSync(process.env.GSD_CONFIG, "utf8"));
        config.version = 1;
        fs.writeFileSync(process.env.GSD_CONFIG, JSON.stringify(config, null, 2));
    ' 2>/dev/null || log_warn "Could not migrate config - please add version: 1 manually"
```

Update the `prepublishOnly` script in package.json to run conflict marker detection before building hooks.

Current:
```json
"prepublishOnly": "npm run build:hooks"
```

Change to:
```json
"prepublishOnly": "git diff --check HEAD && bun run build:hooks"
```

Explanation:
- `git diff --check HEAD` checks all tracked files for conflict markers (<<<<<<, ======, >>>>>>). Exits non-zero if any found.
- This runs BEFORE build:hooks, so a failure blocks the publish entirely.
- Uses `bun run build:hooks` instead of `npm run build:hooks` (tooling preference).

Also update the build:hooks reference to use bun:
```json
"scripts": {
    "install": "node bin/install.js",
    "build:hooks": "node scripts/build-hooks.js",
    "lint": "eslint .",
    "lint:fix": "eslint . --fix",
    "prepublishOnly": "git diff --check HEAD && bun run build:hooks"
}
```

Note: The `lint` and `lint:fix` scripts may have been added by Plan 01. If they are already there, keep them. If not, add them here as well.
